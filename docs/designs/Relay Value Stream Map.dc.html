RELAY · VALUE STREAM MAP

The value stream map

The projection worth building first. This page shows how we derive it from data Relay already keeps, and what it looks like at two scopes — one card, and the last 20 averaged. A note on words: this is a Lean tool, but the reader isn't a Lean expert, so every borrowed term gets a plain-English gloss the first time.

01 — How we get to it

Three steps from the log to the map

Nothing new gets tracked. Relay already writes an event every time a card moves, changes owner, or an agent runs. We fold those events into per-stage spans, then draw the spans as a map.

The event log Relay already keeps

Every card is a timestamped stream of events, each tagged with who did it and (for agent runs) what it cost.

04 — Reading it

What the map tells you, and the lever

THE LONGEST RISE

The tallest waiting rung is your biggest time sink — here, cards parked in Review. Lever: auto-approve low-risk gates, add an SLA nudge.

THE FATTEST $ BOX

The data box with the most spend is your money sink — Code. Lever: cap context, route routine work to a cheaper, faster model.

FLOW EFFICIENCY

The one number to trend. Rising means work flows; falling means waiting is creeping back in. Set a target and watch it weekly.

05 — A lens we're weighing not decided

Splitting the valleys: needed vs over-work

The idea you liked but weren't sure about. A canonical VSM treats all working time as value. This overlay questions that — it floors each valley with necessary vs over-work (the agent grinding an hour where ten minutes would do). It targets our real #1 waste, but it's a departure from the standard map, so treat it as a toggle, not the default.

⚠ The open question: "needed" requires a baseline for what a task type should cost — a median per tag, or a human expectation. Without that, over-work is a guess. Worth prototyping the baseline before we commit to this lens.
