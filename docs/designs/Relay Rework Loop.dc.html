RELAY · POST-DEPLOY REWORK

Handling rework after deploy

A card ships. Weeks later a bug turns up and the work has to go around again — another spec, plan, code, deploy. Our pipeline is built to flow forward once. This doc lays out six ways to model the loop, each with a board view and a card view, and rates every one on the thing you actually care about: how little Relay has to know.

00 — The problem

The pipeline is a line; work is a loop

The board reads left to right and stops. But a shipped feature can come back at any time. The tempting fix — teach Relay to detect "went backward" and spin up a new cycle — forces it to understand which backward move means what. That's exactly the knowledge you don't want it to hold.

Spec → Plan → Code → Deploy → Complete bug found
↩ Now what? It has to travel most of that line again — but the card is already "done," its spec is written, and nobody wants Relay guessing when a rewind is a real re-do.
01 — The principle that dissolves it

Re-entering the pipeline is a human gesture, never an inference

This one rule is shared by all six treatments and is the answer to "Relay shouldn't know too much." Relay never watches for backward movement and decides to start a cycle. Instead, a person reopens the work — presses a button, drops a card on a stage, files a bug — and that gesture places something on an active stage. From there Relay does the only thing it ever does: work whatever is sitting on an active stage it's allowed to touch. The six treatments differ only in what that gesture creates and how it reads.

RELAY KNOWS

Stage types, ownership, and "pull the next thing when I have room." Nothing about features, bugs, or versions.

HUMANS DECIDE

When something re-opens, where it re-enters, and whether it's a small patch or a full re-do.

NO NEW RULES

"Backward from Review" and "backward from Spec" need no special handling — both are just a card landing on a stage.

Treatment A the human model, literally

Reopen

There are no runs, no children, no versions. A Done card has a Reopen button. Pressing it drops the card back onto Spec (or a stage you pick) and it flows forward again — same card, same artifacts, edited in place. History lives in the append-only activity log. This is exactly what a person means by "open it and do it again."

Treatment B Claude Code's idea, de-risked

Runs inside a card

The card is the durable feature. A run is one pass through the pipeline, with its own spec / plan / deploy snapshot. The card detail stacks its runs. The fix for the awkwardness you flagged: a new run is only ever started by a human pressing "Start a fix" — Relay never creates one from a backward move. So Relay still only knows "a card has one active run; work it."

✳ This is the difference from the version that felt off: the run/card split stays, but the trigger is explicit and human, so Relay needs no "if it goes back from Review, branch" logic.

Treatment C your "run kanban + card kanban" idea, drawn out

Two boards: features above, runs below

Split the levels. A coarse feature board (Inbox → In flight → Shipped) is where humans track outcomes. A detailed run board is the pipeline where Relay lives and only ever sees runs. Activating a feature spawns a run below; when the run deploys, the feature moves to Shipped. A bug spawns another run and the feature slides back to In flight. Relay has no concept of the feature layer at all — it's the purest version of "Relay doesn't know too much."

Treatment D one mechanism for decomposition AND rework

Sub-cards

A card can hold child cards, and a child is just a normal card that flows the full pipeline on its own. Splitting a big feature into pieces and filing a post-deploy bug use the exact same move — add a child. The parent is a lightweight container that rolls up its children's status. To Relay every card is identical; parent/child is a link it never has to reason about.

Treatment E the counter-option — no board regression

Revision in place

The one option where the card never moves backward. A shipped card stays parked in Complete. Opening it, you "Start a revision," which runs a compact pipeline inside the card while it holds its place with a quiet "revising" ring. Spec and plan become versioned documents (v1 shipped / v2 draft). Shown here as the deliberate contrast to your pick of "jump back to Spec" — worth seeing what you'd be giving up, and what it costs.

Treatment F runs, but in a metaphor Claude Code users already own

Branches

Treat the card like a repo. Mainline is what's deployed. A bug opens a branch that travels the pipeline and merges (deploys) back to mainline. Card history reads as a branch graph. Mechanically this is Treatment B — the trigger to open a branch is an explicit human act, so Relay just "works the active branch" — but the git framing lands instantly with an engineering audience.

07 — Smarter review stages

Reviews route rejects — the loop as a board rule

Every review is a two-outcome gate: pass pulls forward as normal, reject sends the card to one configured stage. The destination is chosen by a human up front, so this stays inside §01 — Relay never infers a route, it just reads the rule. The backend picks up the rejection comment and replans from there.

◆ Two kinds of review, one behavior. Sub-stage reviews reject to their own parent — the destination is implicit in containment. Top-level review stages reject to a stage you pick in board settings (in practice, always Plan).

09 — Where I'd land

The confusion was never runs vs cards

It was letting Relay infer the loop from board movement. Once reopening is an explicit human gesture (§01), that whole problem evaporates — and you're free to pick the lightest representation that fits how often you actually loop. You said: usually one fix, keep Relay dumb, one mechanism for decomposition and rework, and the human model is "open it and do it again." That points somewhere specific.

THE STACK I'D SHIP
A
Reopen is the default

For the common case — a deploy, occasionally a fix — a plain Reopen drops the card back on Spec and it flows again. Nothing new to learn, and it's literally the human model.

D
Sub-cards when it's bigger than a reopen

When a fix deserves its own tracking, or the work splits into pieces, add a child card. It's the only option that answers your "both" — decomposition and rework through one primitive — and Relay still treats every card identically.

Keep in your back pocket: Runs (B) if you later need a defensible per-ship record — but note A + D already give you most of that history through the activity log and child cards, without the card/run duality that felt off. Two boards (C), revision-in-place (E), and branches (F) all solve real problems, but each is heavier than "usually one fix" warrants today.

One thing to nail down: parent state under Sub-cards. Is a parent "Complete" while a child bug is open, or does an open child pull it back to In progress? My lean — parent stays Complete and just shows an "open children" badge, so a shipped feature reads as shipped while the fix flows on its own. That keeps the board honest without any regression feeling.
