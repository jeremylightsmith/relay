Relay Mobile PLATFORM BRIEF · v0.1 · Jul 2026

What Relay is — and isn't — on a phone

Relay's home is the desktop board, where you do deep work with the AI. Mobile is not a second copy of that. It's a companion for keeping work moving when you're away from your desk — its whole job is to unblock the AI and clear the decisions only you can make.

01 · THE LINE WE'RE DRAWING
MOBILE IS…
✓Where you respond to the AI's requests for input
✓Where you review a finished card and approve or reject it
✓A glance at status — what's working, what's blocked, what's waiting on you
✓The thing that pushes you the moment the AI is blocked on you
MOBILE ISN'T…
✕Authoring specs or plans from scratch
✕Building or restructuring a board — stages, columns, rules
✕Long-form writing or heavy multi-card reorganization
✕Admin, billing, integrations setup — desktop work

Rule of thumb: if it takes more than a minute of typing or more than one screen of layout, it belongs on the web board. Mobile is for the decision, not the production.

02 · THE SURFACES WE'RE KEEPING
1A
Inbox — the front door KEEP

A single "Needs you" queue fed by push. This is where 80% of mobile sessions start and end — clear the decision, close the loop, put the phone away.

1B
Board — the map KEEP

For when you want the fuller picture — one stage at a time, swipe between them, tap into any card. Read-mostly, with the same review actions available inline.

1C
Stream — conversational feed DROP

Cut. It reframes structured work as chat and buries the board's status model. The Inbox already delivers the "AI needs you" moment without pretending work is a conversation.

03 · WHAT YOU CAN ACTUALLY DO

Two primary actions carry the app. Everything else is in service of doing those two with confidence.

PRIMARY
Respond to a request for input

The AI paused and asked a question. You answer — pick an option, type a short reply, or send it back with direction. Work resumes immediately.

PRIMARY
Review a card → approve or reject

The AI finished something. You approve it to move on, or reject with a reason. Neither is a blind tap — both need supporting context first (see §04).

Text entry is voice-first
Typing on a phone is the enemy of a quick decision. Tap the mic and speak your answer — transcribed on-device with Whisper. Type only if you'd rather.
SECONDARY · SUPPORTED, NOT THE POINT
Hand a card off to the AI Move a card between stages Comment / reply Reply by voice Search a board or card Reassign a card
04 · "BEFORE I CAN APPROVE, I NEED TO SEE…"

A reject/approve is only as good as the context behind it. The hard design problem for mobile is surfacing that context without turning the phone into a full IDE. Each of these opens as a focused sheet you can dismiss straight back to the decision.

The spec or plan
What was this card supposed to do? Read the doc the AI worked from.
WEB SHEET
The PR on GitHub
The actual diff — deep-links straight into the GitHub app. It's already great; no reason to rebuild it inside Relay.
OPENS GITHUB APP
Comments — soon with screenshots
The discussion thread on the card. Image attachments need a real mobile viewer (pinch-zoom, full-screen).
WEB + NATIVE VIEWER

Design principle: context opens over the decision and dismisses back to it. You should never lose the "approve / reject" bar by going to look at something.

05 · FULLY NATIVE, OR REUSE THE WEB? — RECOMMENDATION

You asked the right question. The honest answer: neither extreme. A pure webview wrapper feels cheap and can't push you or handle screenshots well; a fully native rebuild duplicates the entire product and doubles every future change.

THE CALL

A hybrid — a thin native shell around reused web views. Native owns everything that must feel instant and system-integrated. The existing live-view rendering delivers everything that's content-dense and changes fast. One product, two rendering layers.

LAYER 1Native shell — Swift / Kotlin
Push notifications Tab bar & navigation Inbox list Approve / reject bar Sheets & in-app browser Screenshot viewer Biometrics / share Voice input · Whisper
LAYER 2Web content — reused live view
Card body Spec & plan docs Board columns (responsive) Comment threads Live app preview
LAYER 0Shared core
One API & data model Same auth session Realtime sync (web ⇄ mobile)

Why it works for Relay specifically: the decision surfaces are small and stable (a queue, an action bar, an answer field) — cheap to build native and worth the polish. The content surfaces are large and change constantly (cards, specs, PRs, comments) — exactly what you don't want to maintain twice. The web views already exist; we're wrapping them, not rebuilding them.

FASTEST TO SHIP
Native inbox + push first, wrapping today's card view. Ships the core loop in weeks, not quarters.
WHERE TO INVEST NATIVE
The action bar, transitions, and the screenshot viewer. These are what make it feel like an app, not a site.
THE ONE RISK
Web views must be genuinely mobile-responsive, or the seams show. Budget design time for the card/spec views specifically.
06 · THE NOTIFICATION MODEL

Push is the product. It's the only reason the app is on the phone at all, so the rules for it are load-bearing — not a settings afterthought.

FIRES ON
A card is ready for your review, the AI asks a question, or someone @mentions you. Nothing else pushes — progress and status never do.
BATCHING
One card = one push. A burst of AI activity collapses into a single “3 cards need you” rather than buzzing three times.
QUIET HRS
Respected by default. Work keeps moving overnight; you wake to a batched digest, not a night of buzzes.
BADGE
The app icon badge = the count of items in Needs you. It hits zero when you're caught up — that's the whole feedback loop.
TAP
A push deep-links straight to the card it's about — never to a generic inbox. One tap from lock screen to decision.
07 · SHIP PLAN

Ship the decision loop first; everything else earns its way in. North-star metric: median time-to-unblock — how fast the AI gets its answer once it's stuck.
