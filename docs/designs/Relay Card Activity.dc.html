RELAY · CARD ACTIVITY — SPEC

Card activity

The AI does a lot on a card, and sometimes it stalls or dies — today that work is invisible until you open the card and squint. Every card gets an incoming stream of activity: a live signal on the collapsed card, a readable history once you open it, and a health read so a dead agent surfaces on its own. This is the committed design — what to build and what it looks like. The live Relay Board is a working reference of every piece here.

01 — The entry

One timestamped event: who · did what · when

Everything is built from one atom. Each entry is tagged with a kind so we can colour it, pick the important ones out of the chatter, and later filter. Entries are stored as one flat, append-only list per card — never grouped at rest; any grouping is derived at render.

Action
Routine chatter — read files, ran a command, edited the spec, pushed a commit.
Move
Card changed stage — claimed, advanced, or bounced back on reject. The backbone of the story.
Decision
A needs-you moment — question asked/answered, approved, rejected with a note.
!Failure
The agent errored, got rate-limited, or the run died. The reason this stream exists.
Heartbeat
A silent "still alive" ping on an interval. Invisible in the UI; it's what makes silence mean something.
Entry {
id uuid
card_id ref
kind "action" | "move" | "decision" | "failure" | "heartbeat"
actor "ai" | user_id
text one line, already human-readable
run_id uuid — the AI session that emitted it (null for human/system)
from, to stage ids — only when kind="move"
inserted_at timestamp — the one source of truth for ordering & health
}

run_id is captured from day one even though runs aren't shown in v1 — so grouping by run later needs no backfill. Amended (RLY-148): heartbeat is not a stored entry kind in the implementation — liveness is the cards.agent_heartbeat_at column, bulk-stamped by the runner every 30s; health reads the later of the newest entry and the heartbeat.

02 — On the board

A one-line log strip, coloured by health

When a card has an active agent, its collapsed card grows one line: health dot · newest entry · relative time. That's enough to know it's alive and roughly where it is, without opening anything. The line replaces the old working · % label (the progress bar stays). Cards with no active agent are unchanged from today.

New entries arrive over the socket and update the strip in place. When a card flips to stale or stopped it also fires the board's existing needs you pulse, so it floats up in triage alongside questions and reviews.

03 — Health

Derived from the newest entry — never stored

Health is a pure function of the activity list, computed at render. The running agent emits a heartbeat on an interval, so an absence of entries is itself a signal.

age = now − newest.inserted_at
if newest.kind == "failure" → stopped rose · show Retry
elif not ai_active → none no strip
elif age > STALE_AFTER → stale amber · reads needs-you
else → live violet · dot pulses
The one number to set — STALE_AFTER. v1 is a single global timeout; start around 3× the heartbeat interval and tune. It can go per-stage later (a test run is minutes, a deploy is longer) without changing anything else. Retry re-dispatches the agent on the same card and appends an action entry — it does not clear the history.
04 — In the card detail

One flat timeline, moves shown inline

An Activity section at the bottom of the detail, after Comments — no tab, no separate panel. Strictly newest-first by inserted_at, never grouped. A move renders as a full-width divider chip rather than a dot — that's how "which stage was this in?" is answered, without ever deciding which stage owns a log. Comments (human) and activity (system) stay separate but adjacent.

05 — Build order

How to land it on the stack

1Entry model + append API. Persist the flat list; emit an entry on every agent action, move, and decision, plus the heartbeat on an interval.
2Health function. Pure, derived from the list; unit-test the four branches. This is the whole "it died" feature.
3Collapsed strip. Add the health-coloured line to the card component; push updates over the LiveView socket and reuse the needs-you pulse for stale/stopped.
4Timeline section. Flat render in the detail: entry dots, move dividers, rose failure rows, and the header health chip + Retry.
5Explicitly later, not v1: a "moves only" filter, a verbose/expand toggle for fine actions, and grouping the same timeline under run_id headers once cards routinely get retried.

Reference implementation: the live Relay Board shows the collapsed strip in all three health states (Code & Deploy columns) and the timeline in the card drawer, including Retry.
